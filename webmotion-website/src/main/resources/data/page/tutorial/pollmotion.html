Pollmotion

Présentation

Ce tutoriel vous présente la création d'une application classique avec une base de données. Nous allons focuser notre attention sur différents points techniques pour vous présenter des patterns à utiliser pour vos futur application.

Pour illustrer cette fonctionnalité, nous allons créer un système de sondage, où un utilisateur crée une question avec des choix, et partage l'url à un groupe de personnes qui eux peuvent voter. Nous allons nous focusser sur les points clés de l'application.

Pour rappel, WebMotion est un framework Java web. Il utilise un fichier de mapping pour décrire la liaison entre le serveur et le client. Pour être précis, ce fichier définit le comportement que doit avoir l'application web en fonction des URI appelées. Il se veut proche du standard Java EE, en reposant sur son API et plus précisément sur l'API Servlet 3.

JQuery est une librairie javascript. Il vous permet de manipuler plus simplement les éléments du DOM en proposant de nombreux utilitaires.

Pour plus de renseignements, vous pouvez consulter respectivement les sites internet http://www.webmotion-framework.org et http://jquery.com/.

Une démo de l'application est disponible ici : http://www.webmotion-framework.org/pollmotion/. et l'ensemble du code source est à l'adresse suivante https://gitorious.org/webmotion/webmotion-examples/trees/master/pollmotion.

Layout

Le layout d'un site est la partie qui se retrouve sur chaque page de votre site, ça peut être un élément graphique comme un menu mais ça peut aussi être les inclusions des fichiers javascript et CSS. WebMotion avec l'extra Sitemesh vous propose une gestion des layouts. Dans notre cas, nous allons besoin d'un seul layout.

Voici les éléments pour activer l'extra SiteMesh : 

[config]
server.listener.class=org.debux.webmotion.sitemesh.SiteMeshListener

[filters]
GET          /*                            SiteMesh.decorate       layout=layout.jsp

Database

Pour la gestion de la base de données, nous allons utiliser l'extra JPA proposé par WebMotion. Pour simplifier son utilisation et avoir une approche REST, nous allons séparer les urls en deux domaines un pour le sondage (/poll) et l'autre pour les vote (/vote). 

[config]
server.listener.class=org.debux.webmotion.jpa.JpaListener

[filters]
*            /poll                         Jpa.tx                  persistenceUnitName=pollmotionDB,entityName=org.debux.webmotion.pollmotion.entity.Poll
*            /poll/*                       Jpa.tx                  persistenceUnitName=pollmotionDB,entityName=org.debux.webmotion.pollmotion.entity.Poll
*            /vote                         Jpa.tx                  persistenceUnitName=pollmotionDB,entityName=org.debux.webmotion.pollmotion.entity.Vote
*            /vote/*                       Jpa.tx                  persistenceUnitName=pollmotionDB,entityName=org.debux.webmotion.pollmotion.entity.Vote

La configuration de la base de donnée est un élément sensible à configurer, il est délicat de laisser les mots de passe dans le fichier persitence.xml. WebMotion propose d'écrire les propriétés de configuration de la section [properties] qui permet une externalisation dans un fichier sur le système. 

[pollmotion.properties]
pollmotionDB.hibernate.dialect=org.hibernate.dialect.H2Dialect
pollmotionDB.hibernate.hbm2ddl.auto=update
pollmotionDB.hibernate.connection.driver_class=org.h2.Driver
pollmotionDB.hibernate.connection.url=jdbc:h2:mem:pollmotion;DB_CLOSE_DELAY=-1
pollmotionDB.hibernate.show_sql=false
pollmotionDB.current_session_context_class=thread

La gestion de la transaction se fait par le filtre avec l'injection du dao dans les services.

Services

Vous avez plusieurs possibilités pour la création des services :

  • Partie web avec les services
  • Injecter une partie service
  • Créer un API REST

Partie web avec les services

Dans ce cas le contrôleur est utilisé comme service et permet de les exposer. Cela permet de développer des applications rapidement sans devoir faire une architecture complexe. Cette solution est très utilile pour du prototypage par exemple. Pollmotion est développé sur ce mode.

Exemple de service : 

    public Render getVotes(EntityManager em,
            String pollId) {
        
        GenericDAO daoPoll = new GenericDAO(em, Poll.class);
        Poll poll = (Poll) daoPoll.find(pollId);
        
        GenericDAO daoVote = new GenericDAO(em, Vote.class);
        List<Vote> votes = daoVote.query("findByPollId", 
                                         Parameters.create().add("poll_id", pollId));
        
        int [] results = new int[poll.getChoices().size()];
        for (Vote vote : votes) {
            
            List<Boolean> values = vote.getVotes();
            int index = 0;
            for (Boolean value : values) {
                if (value) {
                    results[index]++;
                }
                index ++;
            }
        }
        
        return renderView("poll_vote.jsp",
                "poll", poll,
                "votes", votes,
                "results", results);
    }

Nous pouvons constater que le service comporte plusieurs étapes :

  1. La récupération des votes
  2. La mise en forme du résultat
  3. Le retour vers la page

La première étape peut être considérer comme un service métier pouvant être utilisable à plusieurs endroit dans votre application.

Injecter une partie service

L'idée est de centraliser le métier dans des classes externes pour être réutilisé dans le contrôleur. Une des solutions est d'utiliser les injectors de WebMotion pour récupérer une instance du service métier en paramètre ou en variable de classe.

Création du service métier : 

public class ServiceVote {
    
    protected EntityManager em;

    public ServiceVote(EntityManager em) {
        this.em = em;
    }

    public List<Vote> getVotes(String pollId) {
        
        GenericDAO daoPoll = new GenericDAO(em, Poll.class);
        Poll poll = (Poll) daoPoll.find(pollId);
        
        GenericDAO daoVote = new GenericDAO(em, Vote.class);
        List<Vote> votes = daoVote.query("findByPollId", 
                                         Parameters.create().add("poll_id", pollId));

        return votes;
    }
}

Création de l'injecteur : 

public class Listener implements WebMotionServerListener {

    @Override
    public void onStart(Mapping mapping, ServerContext context) {
        context.addInjector(new Injector() {
            @Override
            public Object getValue(Mapping mapping, Call call, String name, Class<?> type, Type generic) {
                EntityManager em = // TODO ...
                if (ServiceVote.class.isAssignableFrom(type)) {
                    return new ServiceVote(em);
                }
                return null;
            }
        });
    }

    @Override
    public void onStop(ServerContext context) {
        ...
    }
}

Utilisation du service dans le contrôler : 

    public Render getVotes(ServiceVote service,
            String pollId) {
        
        List<Vote> votes = service.getVotes(pollId);
        
        int [] results = new int[poll.getChoices().size()];
        for (Vote vote : votes) {
            
            List<Boolean> values = vote.getVotes();
            int index = 0;
            for (Boolean value : values) {
                if (value) {
                    results[index]++;
                }
                index ++;
            }
        }
        
        return renderView("poll_vote.jsp",
                "poll", poll,
                "votes", votes,
                "results", results);
    }

Par l'utilisation de l'injecteur, nous séparons les besoins et permettons la réutilisation du code métiers.

Créer un API REST

Une autre possibilité est d'utiliser une proche REST au niveau du contrôler. Cela necéssite la création d'une application «Single page» en javascript. Il est fortement conseiller d'utiliser un framework JS prèvu à cette effet comme AngularJS. Pour masquer le render utilisé, nous utilisons un default render.

Utilisation d'un render par default vers du JSON : 

[config]
default.render=org.debux.webmotion.server.render.DefaultRender

Exemple d'action : 

    public Render getVotes(ServiceVote service,
            String pollId) {
        
        List<Vote> votes = service.getVotes(pollId);
        
        int [] results = new int[poll.getChoices().size()];
        for (Vote vote : votes) {
            
            List<Boolean> values = vote.getVotes();
            int index = 0;
            for (Boolean value : values) {
                if (value) {
                    results[index]++;
                }
                index ++;
            }
        }
        
        Map<String, Object> values = new HasMap<String, Object>();
        values.put("poll", poll);
        values.put("votes", votes);
        values.put("results", results);
        return values;
    }

POST and GET

Un pattern connu pour les sites est le pattern «POST and GET». Le principe est qu' après une action de modification, il faut rédiriger l'utilisateur sur une page d'affichage pour empêcher l'utilisateur de refaire l'action de modification en rafraîchissement son navigateur.

Exemple de service de modification avec une redirection : 

    public Render createVote(GenericDAO dao,
            String pollId,
            String email,
            String[] votes) {
            
        // Create vote
        // ...

        return renderRedirect("/vote/" + pollId);
    }
